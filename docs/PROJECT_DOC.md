## 1. 挑战与系统目标
**章节导读**：这一章解释“为什么简历筛选难”，以及本项目想解决什么问题。
**内容概要**：痛点来自信息多与标准不稳；目标是把筛选拆成可复核的流程。

传统简历筛选很像“在一大堆材料里找最合适的人”，难点不在于缺信息，而在于信息太多、标准不统一、还要兼顾速度与公平。常见痛点包括：

* **时间不够**：简历数量一多，人工筛选很难跟上。
* **标准不一致**：不同人看同一份简历，结论可能不同。
* **只看硬条件会漏人**：满足年限/技能的人很多，但“做事方式”差别很大。
* **只看感觉又不踏实**：软性判断很重要，但需要更稳定的衡量方式。

本项目的目标是做一个“可解释、可量化、可复用”的筛选流程原型，让HR能：

* 直接把岗位描述（JD）粘贴进系统，得到候选人排序结果
* 同时考虑两类信息：**硬性门槛**（必须满足）与**软性特质**（更像加分项）
* 看到“为什么排在前面/为什么被淘汰”的理由线索，方便复核与沟通

> 【小贴士】什么是“硬性门槛/软性特质”  
> 硬性门槛像“入场券”：不满足就无法进入下一轮，比如最低年限、英语等级、必须技能。  
> 软性特质像“加分项”：能拉开同档候选人的差距，比如沟通、协作、抗压、学习能力。

**核心要点**

* 筛选的难点是“信息多 + 标准不稳 + 需要解释”
* 最实用的方式是把筛选拆成“先过门槛，再比优劣”

## 2. 一句话看懂我们怎么做
**章节导读**：这一章用一个比喻，把全流程讲清楚。
**内容概要**：用“漏斗”分三步完成筛选；输出不仅有排名，还有复核线索。

把筛选想成一个“漏斗”：

* 上面口子大：先把“看起来相关”的人都捞进来（避免漏掉）
* 中间收紧：用硬性门槛把明显不符合的人筛掉（保证底线）
* 最后精挑：用软性特质做加分比较，排出优先联系顺序（提高命中率）

**输入-处理-输出（总览）**

* 输入：一段岗位描述（JD）
* 处理：理解岗位 → 找相关候选 → 过硬门槛 → 软性加分排序
* 输出：候选人列表（含Top推荐、详细名单、关键理由线索）

**流程图（文字版）**

* JD（岗位描述）
  → 系统提炼“必须满足”和“优先偏好”
  → 先找一批“整体方向接近”的候选人
  → 检查硬性门槛（不满足就淘汰）
  → 比较软性特质（满足越多/越贴近，排名越靠前）
  → 输出：Top推荐 + 全量列表 + 每人的关键信息

**核心要点**

* 先“别漏人”，再“别错人”
* 排序结果不仅给结论，也给复核线索

## 3. 系统分工（像一个招聘团队）
**章节导读**：这一章讲“各部分做什么”，强调协作关系，而不是实现细节。
**内容概要**：把系统看成团队分工：整理、翻译、裁判、展示、记忆，串成一条流水线。

你可以把系统理解成一个小团队，分工协作：

### 3.1 候选人信息整理员（把简历变得更好读）
* 输入：候选人的各类文本信息（岗位意向、亮点、更多介绍、简历正文等）
* 处理：把分散内容整理成一份“统一档案”，并提取出关键要点（如技能、年限、教育、英语等级等）
* 输出：每位候选人的“可对比档案”（后续筛选与展示都用它）

### 3.2 岗位需求翻译官（把JD变成可执行的条件）
* 输入：HR粘贴的岗位描述（JD）
* 处理：把JD里的要求拆成两类：必须满足（硬性）与优先偏好（软性）
* 输出：一份“岗位条件清单”（包含硬性门槛与软性特质）

### 3.3 筛选裁判（按规则做淘汰与排序）
* 输入：候选人档案 + 岗位条件清单
* 处理：先检查硬性门槛，再对软性特质做加分比较
* 输出：候选人排序结果，以及每位候选人的关键得分线索

### 3.4 展示与复核助手（把结果讲清楚）
* 输入：排序结果与候选人关键线索
* 处理：把“Top推荐”和“全部候选人”以卡片形式展示，方便点击查看细节
* 输出：可浏览、可对照、可解释的结果页面

### 3.5 记忆本（让系统越用越省事）
* 输入：候选人档案与岗位条件清单的提炼结果
* 处理：把已经做过的“整理/提炼”先记下来，避免重复劳动
* 输出：下一次同样的数据来时更快、更稳定

> 【小贴士】为什么要“先整理再筛选”  
> 原始简历像“各处散落的线索”。先整理成统一档案，相当于把线索装订成册，后面比较时才不会“看漏、看偏”。

**核心要点**

* 系统不是一口气给结论，而是分工协作完成“整理→翻译→裁判→展示”
* “记忆本”让重复工作变少，结果更稳定

## 4. 关键规则
**章节导读**：这一章讲最关键的判定逻辑：硬性怎么判、软性怎么比。
**内容概要**：硬性守底线（年限/英语/必须技能）；软性用“证据”做加分比较。

### 4.1 硬性门槛：不满足就不能进下一轮
硬性门槛主要用来保证“底线正确”。常见包括：

* **经验年限**：例如“至少3年”，不够就淘汰
* **英语等级**：例如“至少upper”，不够就淘汰
* **必须技能**：例如“必须会A/B/C”，缺得太多就淘汰

### 4.2 英语等级：统一成五档，按“从低到高”严格比较
为了避免同一个意思出现多种写法导致混乱，系统把英语能力统一成五个档位（从低到高）：

* basic → pre → intermediate → upper → fluent

判定方式很直观：

* JD写“至少到某一档”，候选人的档位必须**不低于**它才算通过
* JD不写英语要求，则不把英语当作淘汰项

> 【小贴士】“严格比较”是什么意思  
> 不是“差不多就行”，而是像台阶一样：站在更高台阶的人当然能满足更低台阶的要求，但反过来不行。

### 4.3 必须技能：看“覆盖得够不够”
对于必须技能，系统关注的是“该有的有没有、缺得多不多”：

* 如果岗位写了5个必须技能，候选人至少覆盖到一定比例，才算通过
* 这样能避免“看起来相关但关键能力缺失”的误判

### 4.4 软性特质：像“找证据”，从候选经历里挑最贴近的一段
软性特质往往写得短（例如“沟通好、能抗压、推进力强”），直接把它和整份简历硬对比，容易被简历的其他内容“淹没”。

更贴近日常判断的做法是：

* 把软性要求当作若干“提问”
* 在候选人的经历描述里寻找“最能回答这些提问的一段”
* 以这段“最贴近的证据”为准，给出0到1之间的软性匹配分（越接近1表示越贴合）

**核心要点**

* 硬性门槛保证“底线正确”，软性特质负责“拉开差距”
* 英语等级统一成五档，按台阶严格比较
* 软性判断更像“找证据”，而不是只看整体印象

## 5. 怎么用（按步骤走）
**章节导读**：这一章给出非技术使用视角的流程。
**内容概要**：粘贴JD即可得到Top推荐与全量列表，用于行动与复核。

**输入-处理-输出（使用视角）**

* 输入：把岗位JD粘贴到页面输入框
* 处理：系统自动整理候选人档案、提炼岗位条件、做筛选与排序
* 输出：你会看到两块结果
  * Top推荐：优先建议联系的人选
  * 全部候选人：可逐个打开查看关键线索与详情

> 【小贴士】你可以把它当作“第一轮助理筛选”  
> 系统给出的是“优先级建议”和“复核线索”，不替代最终决策，但能把你从重复劳动里解放出来。

**核心要点**

* 只需要提供JD，其余整理与筛选由系统自动完成
* Top推荐用于快速行动，详细列表用于复核与对比

## 6. 项目案例：招聘数据工程师
**章节导读**：这一章用一个具体岗位，演示“硬性+软性”如何协作。
**内容概要**：用“数据工程师”岗位演示：先过门槛，再看证据，最后排序呈现。

### 6.1 输入（JD示例，用人话）
假设你要招一位数据工程师，JD大意是：

* 必须满足：
  * 经验至少3年
  * 英语至少upper
  * 必须技能包含：数据处理、数据管道（让数据自动流转的工作流程）、至少一种主流数据工具（例如Python：常用编程语言；SQL：常用数据查询语言；Spark：常见的大规模数据处理工具）
* 优先偏好：
  * 跨团队沟通顺畅
  * 能把事情推进到落地
  * 文档习惯好、表达清晰

### 6.2 处理（系统会怎么做）
* 先把每位候选人的信息整理成“统一档案”
* 从JD里提炼出“硬性门槛”和“软性偏好”
* 先做初步相关性筛选（先把方向相近的人捞进来）
* 再过硬性门槛（年限、英语、必须技能覆盖）
* 最后比较软性特质（从经历里找最贴近的证据，给出加分）

### 6.3 输出（HR会看到什么）
* Top推荐里的人通常具备两个特点：
  * 硬性门槛扎实通过
  * 软性证据更充分（例如明确写过跨团队协作、推动项目上线、沉淀文档等）
* 详细列表里，你能快速定位差距在哪里：
  * 有的人被淘汰是因为硬性缺口（例如英语不足或关键技能缺得多）
  * 有的人硬性通过但软性分不高（例如经历描述里缺少“推进/协作”的证据）

**核心要点**

* 先用硬性门槛守住底线，再用软性证据拉开差距
* 输出结果既能“快”（Top推荐），也能“稳”（可复核的线索）

## 7. 常见疑问
**章节导读**：这一章回答使用时最常见的困惑。
**内容概要**：解释“为何排名低/为何淘汰/英语怎么比/结果如何使用”等问题。

* **我觉得某个候选人不错，但排名不高？**
  * 常见原因是软性证据在文本里不明显。系统更偏向“写得清楚、证据充分”的候选人，建议结合你的经验做二次复核。
* **为什么有人被直接淘汰？**
  * 多数是硬性门槛未达标（例如年限、英语等级、必须技能覆盖不足）。这类淘汰是为了保证后续比较的基础一致。
* **英语等级是怎么理解的？**
  * 统一成五档并按台阶比较：basic、pre、intermediate、upper、fluent。岗位写“至少upper”，则upper和fluent都算通过，intermediate不通过。
* **系统给的是最终结论吗？**
  * 更像“第一轮助理筛选”：提供优先级建议与复核线索，最终决策仍由HR和用人团队把关。

**核心要点**

* 排名和淘汰都能回到“门槛是否通过、证据是否充分”来理解
* 系统负责提效与提供复核线索，最终判断仍由人来做
